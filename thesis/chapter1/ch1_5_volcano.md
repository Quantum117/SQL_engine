# 1.7. Модель Volcano (Iterator Model): Первый способ исполнения

После того как оптимизатор построил логический план, встаёт вопрос о **физическом исполнении**: как движок итерирует реальные данные из CSV-файлов и вычисляет итоговый результат?

### 1.7.1. Принцип работы Volcano

Модель Volcano, предложенная Гётцем Грэфе в 1994 году, на протяжении десятилетий оставалась стандартом для реляционных СУБД (PostgreSQL, MySQL). В этой модели **каждый оператор реализует интерфейс итератора** с методом `__next__()`.

Верхний узел дерева запрашивает строку у дочернего, тот — у своего, и так далее до `Scan`. Это называется **pull-based** (вытягивающая) модель.

### 1.7.2. Трассировка вызовов для сквозного примера

Покажем, что происходит при вычислении нашего запроса в Volcano-модели. Физическое дерево после оптимизации:

```
Aggregate(SUM salary, GROUP BY name)
  └── Filter(dept_id = 10)
        └── Scan(employees.csv)
```

Каждый вызов `result.__next__()` порождает цепочку:

```
1. Aggregate.__next__()
      → вызывает Filter.__next__()
           → вызывает Scan.__next__()   ← читает строку 1 из CSV
      → строка не удовлетворяет dept_id=10, повтор...
           → вызывает Scan.__next__()   ← читает строку 2 из CSV
      → ...продолжается до первой строки с dept_id=10
      → возвращает строку наверх в Aggregate
2. Aggregate накапливает строки группы, снова вызывает Filter.__next__()
   ...
N. Scan.__next__() бросает StopIteration — данные кончились
N+1. Aggregate флашит накопленные группы → возвращает финальный результат
```

Для таблицы с 1 000 000 строк это **1 000 000 вызовов** `Filter.__next__()` и 1 000 000 вызовов `Scan.__next__()`, даже после оптимизации.

### 1.7.3. Проблема «Memory Wall» и задержки кэша

Современные процессоры работают значительно быстрее, чем оперативная память. Это создаёт проблему «Memory Wall» (стена памяти):

| Уровень | Задержка |
|:--------|:---------|
| L1 Cache | ~1–4 такта |
| L2 Cache | ~10–12 тактов |
| L3 Cache | ~30–70 тактов |
| RAM | ~200–300 тактов |

В Volcano-модели при каждом вызове `__next__()` процессор обрабатывает **одну строку** — словарь Python. Строка не обеспечивает достаточной локальности данных для заполнения кэш-линий, поэтому CPU вынужден постоянно обращаться к RAM. Это приводит к критическим простоям.

### 1.7.4. Проблема предсказания переходов (Branch Prediction)

В Python вызов `Filter.__next__()` — это **виртуальный вызов**: интерпретатор не знает заранее, какой именно объект стоит за этим методом. Процессор пытается предсказать следующую инструкцию, но постоянное «перепрыгивание» между узлами дерева (Scan → Filter → Aggregate) вызывает **ошибки предсказания** и сброс конвейера процессора.

### 1.7.5. Преимущество Volcano: Минимальная память

Volcano не лишён достоинств. Главное из них — **минимальное потребление RAM**: в каждый момент времени в памяти присутствует не более одной строки (кроме буфера агрегата). Для OLTP-нагрузок (чтение/запись одной записи пользователя) это идеальная модель.

---

> **→ Переход к разделу 1.8.** Volcano — корректная, но медленная для аналитики модель. Чтобы задействовать возможности современного iron (SIMD, кэш-линии L1/L2), нужна принципиально другая архитектура — **векторизованное исполнение**.
