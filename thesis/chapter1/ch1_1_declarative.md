
# 1.1. SQL как декларативный язык программирования и реляционная модель

> **Сквозной пример.** На протяжении всей первой главы мы будем шаг за шагом разбирать один демонстрационный запрос, прослеживая его путь от строки текста до числа на экране:
> ```sql
> SELECT name, SUM(salary)
> FROM employees
> WHERE dept_id = 10
> GROUP BY name
> ORDER BY SUM(salary) DESC;
> ```
> К концу главы станет ясно, почему для исполнения этого запроса требуется целый движок — а не просто цикл `for` на Python.

### 1.1.1. Парадигмальный сдвиг: От навигации к реляционной логике

До начала 1970-х годов программирование баз данных носило исключительно **императивный (навигационный)** характер. Разработчик должен был вручную описывать алгоритм обхода указателей на диске — по сути, управлять «железом» напрямую, как в языке ассемблера.

В 1970 году Эдгар Кодд в работе «A Relational Model of Data for Large Shared Data Banks» предложил радикальную идею: рассматривать данные как **математические отношения** (множества), а взаимодействие с ними — как логические операции над предикатами первого порядка.

Этот переход сопоставим с переходом от ассемблера к языкам высокого уровня: программист перестал управлять физическим расположением данных и перешёл к описанию абстрактных логических связей.

 Проект System R и становление синтаксиса

Разработанный в IBM язык **SEQUEL** (позже SQL) стал реализацией идей Кодда. Важно понимать, что SQL изначально проектировался не как утилита, а как **высокоуровневый специализированный язык (DSL)**. Его «английский» синтаксис был попыткой сделать программирование логики данных доступным, сохраняя при этом строгость реляционной алгебры.

### 1.1.3. Вычислительная мощность и полнота по Тьюрингу

Эволюция стандартов превратила SQL из простого языка запросов в мощный инструмент вычислений:

1. **SQL-86 и SQL-92**: Оформили ядро языка, реализующее реляционную алгебру.
2. **SQL:1999 (SQL3)**: Критическая точка в истории. С введением **рекурсивных общих табличных выражений (Recursive CTE)** SQL официально стал **полным по Тьюрингу**. Это означает, что на SQL можно реализовать любой алгоритм, поддающийся вычислению.
3. **Современные стандарты (2011–2023)**: Добавили поддержку JSON/XML, временных рядов и аналитических оконных функций, де-факто сделав SQL основным языком для Data Engineering.

### 1.1.4. Декларативная архитектура и компиляция запросов

Ключевое отличие SQL как языка программирования — полная изоляция **логики** от **реализации**. В императивном программировании (Python, C++) код определяет путь исполнения. В SQL программа представляет собой описание желаемого состояния — **Abstract Syntax Tree (AST)**, которое передаётся **оптимизатору** для самостоятельного выбора физического плана.

| Характеристика | Императивное программирование | Декларативное программирование (SQL) |
| :--- | :--- | :--- |
| **Единица описания** | Алгоритм (последовательность шагов) | Отношение (логическое выражение) |
| **Исполнение** | Прямая интерпретация / компиляция | Трансформация в физический план |
| **Оптимизация** | Ручная (выбор структур данных) | Автоматическая (Cost-Based Optimization) |
| **Тьюринг-полнота** | Прямая (циклы, рекурсия) | Через Recursive CTE и хранимые процедуры |

В рамках данной работы мы рассматриваем SQL не просто как средство извлечения данных, а как **целевую вычислительную среду**. Наша задача — реализовать ядро (Engine), которое транслирует декларативные конструкции языка в эффективный план выполнения, сопоставимый по производительности с императивными алгоритмами обработки данных.
