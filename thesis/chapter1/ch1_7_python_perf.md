# 1.9. Python как инструментальная среда: Обоснование выбора

Создание высокопроизводительного программного обеспечения на Python часто воспринимается как противоречие. Данный раздел объясняет, почему Python является **оправданным выбором** для реализации исследовательского SQL-движка, и где именно производительность обеспечивается не самим Python, а нативными библиотеками.

### 1.9.1. Две причины медленности CPython

Стандартный интерпретатор Python (CPython) имеет две фундаментальные причины производительных проблем:

1. **Динамическая типизация и боксинг.** Каждое число Python — это полноценный объект-«коробка» (boxing) с заголовком из ~28 байт: счётчиком ссылок, указателем на тип и самим значением. Операция `a + b` требует: распаковки `a`, распаковки `b`, создания нового объекта-результата. Для числовых расчётов над миллионом строк это катастрофически дорого.

2. **Global Interpreter Lock (GIL).** CPython ограничивает выполнение байткода **одним потоком** одновременно, делая стандартные `threading` неэффективными для CPU-задач.

### 1.9.2. Решение: Граница Python / NumPy как архитектурный шов

Ключевая идея нашего движка — **Python управляет, NumPy вычисляет**. Python-код отвечает за:
- Управление жизненным циклом операторов.
- Координацию батчей между узлами дерева.
- Логику ветвления (какой оператор вызвать следующим).

NumPy (написанный на C и Fortran) отвечает за:
- Все арифметические операции над массивами (`sum`, `mean`, `*`, `+`).
- Фильтрацию булевыми масками (`arr[mask]`).
- Сортировку (`np.argsort`).

Фактически Python-интерпретатор исполняет ~977 высокоуровневых вызовов (количество батчей), а вся арифметика происходит внутри скомпилированного C-кода NumPy — без динамической типизации и без GIL на уровне математики.

### 1.9.3. Почему Python — правильный выбор для исследовательского движка

| Критерий | Python + NumPy | C++ / Rust |
|:---------|:---------------|:-----------|
| **Скорость разработки** | Высокая (DSL-like код, REPL) | Низкая (компиляция, управление памятью) |
| **Скорость исполнения** | ~3–15× медленнее нативного | Максимальная |
| **Интроспекция и визуализация** | Встроенная (Matplotlib, Graphviz) | Требует внешних инструментов |
| **Читаемость для проверяющего** | Высокая | Средняя |
| **Целевая нагрузка** | OLAP-аналитика (batch) | OLTP (1 запрос / мс) |

Для **исследовательского** движка, цель которого — продемонстрировать алгоритмы и архитектурные решения, Python является стандартным выбором (так устроен Apache Spark PySpark, Apache Arrow, и большинство исследовательских СУБД). Производственную нагрузку движок не несёт.
