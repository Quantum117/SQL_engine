# 1.8. Векторизованная модель исполнения: Современный подход

Векторизация — это ответ на проблемы медленной интерпретации и ограниченного использования ресурсов современных процессоров. Вместо обработки **одной строки за раз** операторы обмениваются **пакетами (batch)** данных.

### 1.8.1. Принцип пакетной обработки

В нашем движке каждый физический оператор реализует метод `get_next_batch()`, возвращающий не одну строку, а **словарь NumPy-массивов** — по одному массиву на колонку:

```python
# Volcano: один словарь — одна строка
{"name": "Alice", "salary": 85000, "dept_id": 10}

# Vectorized: один словарь — 1024 строки (batch)
{
  "name":    np.array(["Alice", "Bob", ..., "Zara"]),    # 1024 элемента
  "salary":  np.array([85000, 92000, ..., 78000]),       # 1024 элемента
  "dept_id": np.array([10, 10, ..., 10])                 # 1024 элемента
}
```

### 1.8.2. Применение к сквозному примеру: Сравнение с Volcano

Для нашего запроса с 1 000 000 строк:

| Метрика | Volcano | Vectorized (batch=1024) |
|:--------|:--------|:------------------------|
| Вызовов `__next__()` / `get_next_batch()` | 1 000 000 | ~977 |
| Проверок типов Python на строку | 1 | 1 (на весь batch) |
| Фильтрация | `if row["dept_id"] == 10` × 1M | `arr[arr["dept_id"] == 10]` (NumPy) |
| Использование L1/L2 кэша | Низкое (1 строка за раз) | Высокое (массив ~8KB помещается в L1) |

При размере пакета 1024 строки количество вызовов функций сокращается в **1024 раза**. Проверка типов и интерпретация Python-байткода происходит один раз на весь пакет.

### 1.8.3. Эффективность кэша процессора (Cache Efficiency)

Данные в массивах NumPy хранятся в памяти **плотно и последовательно** (contiguous layout). Когда процессор читает первый элемент массива `salary`, он автоматически подгружает в L1-кэш следующие ~15 соседних значений (одна кэш-линия = 64 байта = 8 числа типа `float64`). Следующий элемент берётся прямо из кэша — **без обращения к RAM**.

### 1.8.4. Построчная vs. Колоночная компоновка (Row vs. Columnar)

Для понимания эффективности векторизации важно рассмотреть два способа компоновки данных в памяти:

1. **Row-major (NSM — N-ary Storage Model)**: данные одной строки хранятся рядом. Идеально для OLTP (транзакций): быстро считать/записать всего одного пользователя.
2. **Columnar (DSM — Decomposition Storage Model)**: данные одной колонки хранятся рядом. Идеально для OLAP (аналитики): просуммировать только `salary` для миллиона строк — читаем только один столбец.

Наш векторизованный движок использует **колоночный подход**. При агрегации `SUM(salary)` процессор читает только массив `salary`, не тратя пропускную способность шины памяти на `name`, `dept_id` и другие неиспользуемые атрибуты.

### 1.8.5. SIMD (Single Instruction, Multiple Data)

Векторизованное исполнение через NumPy позволяет задействовать **SIMD-инструкции процессора** (SSE, AVX). Это аппаратная возможность CPU выполнить одну операцию одновременно над несколькими парами данных за один такт. Например, AVX2 складывает **4 пары double** за один такт вместо одной. Для нашего `SUM(salary)` это означает 4× аппаратное ускорение на каждую кэш-линию.
