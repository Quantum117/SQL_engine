# 1.3. Лексический анализ: Преобразование текста в поток токенов

Первым этапом трансляции является лексический анализ (сканирование, токенизация). Его задача — преобразовать линейную последовательность символов запроса в структурированный поток логических единиц — **токенов**.

### 1.3.1. Понятия лексемы, токена и паттерна

Для понимания работы сканера необходимо различать три базовых понятия:

1. **Лексема (Lexeme)** — фактическая последовательность символов в исходном тексте (например, `SELECT`, `123.45`, `dept_id`).
2. **Токен (Token)** — объект, описывающий лексему. Он содержит *тип* (категорию) и *атрибутивное значение*.
3. **Паттерн (Pattern)** — правило, описывающее множество лексем, образующих данный тип токена. Паттерны задаются **регулярными выражениями**.

### 1.3.2. Применение к сквозному примеру: Полная таблица токенов

Покажем, как лексер разбивает наш демонстрационный запрос на токены:

```
SELECT name, SUM(salary) FROM employees WHERE dept_id = 10 GROUP BY name ORDER BY SUM(salary) DESC;
```

| № | Лексема | Тип токена | Примечание |
|:--|:--------|:-----------|:-----------|
| 1 | `SELECT` | `KW_SELECT` | Ключевое слово |
| 2 | `name` | `IDENTIFIER` | Имя колонки |
| 3 | `,` | `COMMA` | Разделитель |
| 4 | `SUM` | `IDENTIFIER` | Агрегатная функция |
| 5 | `(` | `LPAR` | Открывающая скобка |
| 6 | `salary` | `IDENTIFIER` | Имя колонки |
| 7 | `)` | `RPAR` | Закрывающая скобка |
| 8 | `FROM` | `KW_FROM` | Ключевое слово |
| 9 | `employees` | `IDENTIFIER` | Имя таблицы |
| 10 | `WHERE` | `KW_WHERE` | Ключевое слово |
| 11 | `dept_id` | `IDENTIFIER` | Имя колонки |
| 12 | `=` | `EQ` | Оператор сравнения |
| 13 | `10` | `NUMBER` | Числовой литерал |
| 14 | `GROUP` | `KW_GROUP` | Ключевое слово |
| 15 | `BY` | `KW_BY` | Ключевое слово |
| 16 | `name` | `IDENTIFIER` | Имя колонки |
| 17 | `ORDER` | `KW_ORDER` | Ключевое слово |
| 18 | `BY` | `KW_BY` | Ключевое слово |
| 19–22 | `SUM(salary)` | _(повтор 4–7)_ | Выражение в ORDER BY |
| 23 | `DESC` | `KW_DESC` | Направление сортировки |
| 24 | `;` | `SEMICOLON` | Конец запроса |

Лексер произвёл **24 токена** из 91 символа. Именно этот поток токенов, а не исходная строка, передаётся дальше на вход парсеру.

### 1.3.3. Архитектура лексера и регулярные выражения

В основе лексера лежат регулярные выражения, описывающие паттерны каждого типа токенов. С математической точки зрения, каждое регулярное выражение преобразуется в **Детерминированный Конечный Автомат (ДКА)**.

Процесс сканирования — это движение автомата по состояниям: начав с $S_0$, лексер читает строку посимвольно. Каждый символ переводит автомат в новое состояние. Если автомат оказывается в «принимающем» состоянии, а следующий символ не позволяет продолжать движение, текущая последовательность признаётся лексемой.

### 1.3.4. Алгоритмический принцип: Longest Match

Лексер применяет механизм **Longest Match (правило максимального совпадения)**: если одна и та же последовательность символов подходит под два правила, лексер выбирает то, которое поглощает больше символов. Например, `GROUP` не будет разбит на `GRO` + `UP` — будет создан один токен `KW_GROUP`.

### 1.3.5. Лексическая спецификация в Lark

В нашем проекте мы используем библиотеку **Lark**, что позволяет описывать лексику декларативно. Вместо ручного написания регулярных выражений на Python, терминалы определяются в файле грамматики `parser/grammar.lark`:

```lark
// Определение терминалов (токенов)
SELECT: "SELECT"i     // 'i' означает регистронезависимость
FROM:   "FROM"i
WHERE:  "WHERE"i
GROUP:  "GROUP"i
BY:     "BY"i
ORDER:  "ORDER"i
DESC:   "DESC"i

IDENTIFIER: /[a-zA-Z_][a-zA-Z0-9_]*/
NUMBER:     /-?\d+(\.\d+)?/
```

### 1.3.6. Очистка потока и обработка «шума»

Лексер также выполняет функцию очистки данных: он удаляет «синтаксический шум» — пробельные символы, переносы строк и комментарии. В Lark это реализуется директивой `%ignore`, которая отсекает лишнее ещё на этапе сканирования.

### 1.3.7. Обработка ошибок и интеграция с таблицей символов

Если сканер встречает символ, не соответствующий ни одному паттерну (например, спецсимволы вне строковых литералов), возникает **лексическая ошибка** с точными координатами (строка и колонка) для диагностики.

Для оптимизации производительности повторяющиеся идентификаторы (такие как `name` в позициях 2 и 16 нашего запроса) хранятся в **Таблице символов** как числовые индексы. Вместо двух строковых объектов `"name"` в памяти будет один объект и два указателя на него — это ускоряет все последующие этапы сравнения.
