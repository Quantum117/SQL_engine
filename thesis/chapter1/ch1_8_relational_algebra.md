# 1.5. Реляционная алгебра как математический фундамент оптимизации

Любой декларативный запрос на языке SQL может быть декомпозирован в выражение реляционной алгебры — формальной системы, оперирующей отношениями. Знание правил этой системы позволяет оптимизатору СУБД **доказывать корректность** своих трансформаций, а не просто применять их эвристически.

### 1.5.1. Базовые реляционные операции

Для математического описания работы нашего движка необходимо выделить расширенный набор операций реляционной алгебры:

1. **Селекция ($\sigma_{c}(R)$)**: Выборка подмножества кортежей из отношения $R$, удовлетворяющих предикату $c$.
2. **Проекция ($\pi_{L}(R)$)**: Уменьшение размерности отношения — оставляем только атрибуты из списка $L$.
3. **Переименование ($\rho_{S}(R)$)**: Изменение имён атрибутов для предотвращения конфликтов при соединениях.
4. **Декартово произведение ($R \times S$)**: Все возможные комбинации кортежей двух отношений.
5. **Тета-соединение ($R \bowtie_{\theta} S$)**: Эквивалентно $\sigma_{\theta}(R \times S)$, где $\theta$ — условие сравнения атрибутов.
6. **Группировка и агрегация ($\gamma_{G; F}(R)$)**: Разбивка отношения на группы по атрибутам $G$ и вычисление агрегатных функций $F$.

### 1.5.2. Применение к сквозному примеру: SQL → Реляционная алгебра

Трансформируем наш демонстрационный запрос в выражение реляционной алгебры. Именно это выражение строит **логический планировщик** (`planner/logical_operators.py`) на основе AST из раздела 1.4:

$$\pi_{\text{name},\, \text{SUM(salary)}} \Bigl( \tau_{\text{SUM(salary) DESC}} \bigl( \gamma_{\text{name};\, \text{SUM(salary)}} \bigl( \sigma_{\text{dept\_id} = 10}(\text{employees}) \bigr) \bigr) \Bigr)$$

Расшифровка слоёв (внутрь → наружу):

| Уровень | Операция | Соответствует |
|:--------|:---------|:--------------|
| Вн. | $\sigma_{\text{dept\_id}=10}(\text{employees})$ | `WHERE dept_id = 10` |
| 2 | $\gamma_{\text{name};\, \text{SUM(salary)}}(\ldots)$ | `GROUP BY name` + `SUM(salary)` |
| 3 | $\tau_{\text{SUM(salary) DESC}}(\ldots)$ | `ORDER BY SUM(salary) DESC` |
| Вн. | $\pi_{\text{name},\, \text{SUM(salary)}}(\ldots)$ | `SELECT name, SUM(salary)` |

### 1.5.3. Реляционное исчисление и полнота языка

Язык SQL считается **реляционно полным**, если он способен выразить любую операцию реляционной алгебры. Наш движок реализует ядро этой полноты, что позволяет гарантировать корректность обработки любых запросов в рамках выбранного подмножества стандарта SQL-92.

### 1.5.4. Законы алгебраической эквивалентности

Математический фундамент позволяет использовать законы преобразования выражений для поиска оптимального плана. Приведём два ключевых закона, реализованных в нашем оптимизаторе:

- **Коммутативность селекции**:  
  $\sigma_{c_1}(\sigma_{c_2}(R)) = \sigma_{c_2}(\sigma_{c_1}(R))$ — порядок фильтров можно менять без изменения результата.

- **Проталкивание селекции через соединение (Predicate Pushdown)**:  
  Если предикат $c$ относится только к атрибутам $R$, то:
  $$\sigma_c(R \bowtie S) = \sigma_c(R) \bowtie S$$
  Это ключевое правило: вместо того чтобы фильтровать результат JOIN (возможно, миллионы строк), мы фильтруем $R$ **до** JOIN — и уменьшаем объём обрабатываемых данных на порядки.

- **Ассоциативность соединений**:  
  $(R \bowtie S) \bowtie T = R \bowtie (S \bowtie T)$ — оптимизатор вправе менять порядок соединения таблиц.

> **→ Переход к разделу 1.6.** Мы знаем **эквивалентные** планы (их могут быть десятки). Как выбрать **оптимальный**? Для этого существует логический оптимизатор, применяющий правила алгебраических эквивалентностей в нужном порядке.
