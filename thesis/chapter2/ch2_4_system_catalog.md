# 2.4. Системный каталог и управление схемами данных

Абстрактное синтаксическое дерево, построенное парсером, структурно корректно, но семантически пусто. Движок знает, что пользователь запросил `SELECT id FROM users`, но не имеет понятия о том, существует ли таблица `users` и есть ли в ней столбец `id`. 

Чтобы превратить синтаксис в исполняемый логический план, системе необходим "словарь", описывающий внешний мир (файлы и структуры данных). Эту роль выполняет **Системный каталог**.

### 2.4.1. Анализ кода: Инкапсуляция метаданных (`TableSchema`)

Метаданные отделены от самих файлов с данными. Вся информация о структуре инкапсулирована в `TableSchema` (файл `catalog.py`). Это позволяет движку производить валидацию запроса *вообще не читая данные с диска*.

```python
class TableSchema:
    def __init__(self, name: str, columns: Dict[str, str], file_path: Optional[str] = None):
        self.name = name
        self.columns = columns         # Маппинг: имя столбца -> тип
        self.file_path = file_path     # Физическое расположение (напр. 'data.csv')
        self.indexes: Dict[str, str] = {} # Метаданные индексов: column_name -> 'HASH'
```

*Интуиция кода*: Архитектурно важно свойство `file_path`. Благодаря такому маппингу, логическое имя таблицы в SQL-запросе (например, `orders`) полностью отвязано от физической реализации хранения (это может быть `csv`, оперативная память, или сетевой сокет). Планировщик оперирует абстрактной схемой, а физический парсер позже обращается к `file_path`.

### 2.4.2. Механизм реестра (`Catalog`)

Сам Каталог — это простой in-memory реестр. В промышленных базах данных (PostgreSQL) каталог сам хранится в таблицах, к которым оптимизатор делает транзакционные запросы, однако для нашей архитектуры достаточен подход потокобезопасного словаря.

```python
class Catalog:
    def __init__(self):
        self.tables: Dict[str, TableSchema] = {}

    def get_table(self, name: str) -> TableSchema:
        if name not in self.tables:
            raise ValueError(f"Table {name} not found in catalog")
        return self.tables[name]
```

### 2.4.3. Роль индексов в каталоге
Особое внимание стоит уделить словарю `self.indexes`, внедренному в схемы таблиц. Когда пользователь регистрирует индекс:
```python
users_schema.add_index("id", index_type="HASH")
catalog.register_table(users_schema)
```
Система немедленно делает этот факт "видимым" для оптимизатора. В последующих фазах (которые будут описаны в Разделе 2.7), Rule-Based Optimizer заглянет в метаданные каталога, увидит доступный `'HASH'` индекс на колонку `id`, и трансформирует оператор линейного сканирования (`SeqScan`) в `IndexScan`, снизив алгоритмическую сложность $O(N)$ до $O(1)$.
