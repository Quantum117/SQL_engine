# 2.2. Спецификация грамматики EBNF и интеграция с Lark

Парсинг — это процесс превращения плоской строки текста в осмысленную иерархическую структуру. В нашем решении лексический и синтаксический анализ не пишется вручную (через тяжелые циклы `while`), а делегируется библиотеке **Lark** через декларативную грамматику EBNF.

### 2.2.1. Интуиция выбора: LALR(1) против Earley
Фреймворк Lark поддерживает два основных алгоритма разбора:
1. **Earley**: Может распарсить абсолютно любую контекстно-свободную грамматику, но работает медленно: сложность достигает $O(N^3)$ в худшем случае.
2. **LALR(1)**: Требует строгой спецификации без неоднозначностей, но генерирует детерминированный конечный автомат, работающий за линейное время $O(N)$. 

Поскольку производительность движка критична, а SQL является достаточно строгим языком, мы проектируем грамматику так, чтобы она компилировалась быстрым парсером LALR(1).

### 2.2.2. Архитектура грамматики (grammar.lark)
Главная хитрость парсинга — это **управление приоритетами операций** без явного программирования. В Lark это достигается вложенностью (иерархией) правил.

Посмотрим на этот ключевой фрагмент `grammar.lark`, описывающий арифметику и логику:

```lark
?expression: logical_or
?logical_or: logical_and (OR logical_and)*
?logical_and: comparison (AND comparison)*
?comparison: arithmetic_expr (OPERATOR arithmetic_expr)*
?arithmetic_expr: term (ADD_OP term)*
?term: factor (MULT_OP factor)*
```
*Анализ кода:* Обратите внимание на инлайн-правила (префикс `?`). Парсер всегда пытается разрешить операцию с наименьшим приоритетом наверху (`OR`), затем спускается к `AND`, затем к `comparison` (`=, >, <`), и только в самом низу разрешает умножение (`MULT_OP`). Если в запросе встречается выражение `price * 2 > 100`, парсер математически гарантированно сначала выполнит `price * 2` (так как правило `term` лежит глубже `comparison`), формируя корректное дерево вычислений.

### 2.2.3. Интеграция парсера в Python (SQLParser)

Связующим звеном между EBNF-файлом и питоновским кодом выступает класс `SQLParser` (в модуле `parser.py`):

```python
class SQLParser:
    def __init__(self):
        grammar_path = os.path.join(os.path.dirname(__file__), 'grammar.lark')
        with open(grammar_path, 'r') as f:
            self.grammar = f.read()
            
        # Компилируем грамматику алгоритмом LALR(1)
        self.lark = Lark(self.grammar, start='select_statement', parser='lalr')
        self.transformer = SQLTransformer()

    def parse(self, sql_text: str):
        # 1. Токенизация и построение дерева разбора
        tree = self.lark.parse(sql_text)
        # 2. Трансформация дерева разбора в наше кастомное AST
        return self.transformer.transform(tree)
```

В методе `parse` мы видим, что выход библиотеки (`tree`) не используется напрямую. Он прокидывается в объект `SQLTransformer`. Почему? Потому что нативное "Parse Tree" (дерево разбора) излишне загромождено. Оно хранит каждую запятую, скобки и служебные токены. Дальнейшая архитектура движка требует чистого Абстрактного Синтаксического Дерева (AST), процесс формирования которого будет разобран в следующем разделе.
