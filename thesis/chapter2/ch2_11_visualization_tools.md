# 2.11. Инструменты визуализации планов и бенчмаркинга

Разработка СУБД сопряжена с отладкой чрезвычайно абстрактных структур: деревьев и графов. Отлаживать их математические трансформации с помощью обычного текстового вывода (команды `print`) невозможно. Для решения этой проблемы была разработана подсистема визуализации и бенчмаркинга, придерживающаяся концепции "Белого ящика" (White-box).

### 2.11.1. Визуальный аудит через Graphviz (DOT)
Для отладки Rule-Based Optimizer (RBO) критически важно видеть разницу между изначальным и оптимизированным планом. Модуль `visualization/visualizer.py` использует библиотеку Graphviz. У каждого логического оператора реализован метод трансляции своей семантики в формат `.dot`:

*Код генерации узлов графа*:
```python
def to_dot(self, dot, parent_id=None):
    node_id = str(id(self))
    # Формируем метку узла с данными о пушенных предикатах
    label = f"Filter\\nCondition: {self.condition.pretty()}"
    dot.node(node_id, label, shape="diamond", style="filled", fillcolor="lightyellow")
    
    if parent_id:
        dot.edge(parent_id, node_id)
        
    # Рекурсивный проход вглубь плана
    self.input.to_dot(dot, node_id)
```
*Интуиция кода*: Применяя функцию `visualizer.visualize(plan)`, разработчик получает PNG-файл, в котором узлы покрашены в разные цвета (например, `JOIN` в синий, `SCAN` в зеленый). Если правило `Predicate Pushdown` (описанное в 2.7) сработало правильно, то на графе визуально видно, как желтый ромб фильтра "спустился" ниже синего узла `JOIN`.

### 2.11.2. CLI-Инструментарий
Для ускорения разработки и интеграции в CI/CD (или серверные среды, где нет графического интерфейса), реализован консольный отладчик. Он строит ASCII-дерево прямо в терминале:

```text
|-- LogicalProject (Columns: user_id, amount)
    |-- LogicalFilter (Condition: amount > 1000)
        |-- LogicalJoin (INNER JOIN ON u.id = o.user_id)
            |-- LogicalScan (Table: users)
            |-- LogicalScan (Table: orders)
```

### 2.11.3. Архитектура Бенчмаркинга
Модуль `benchmarks/` не просто замеряет время выполнения программы с помощью секундомера. Он глубоко внедрен в физические операторы. Если вы обратили внимание на базовые классы в Разделах 2.8 и 2.9, каждый оператор инстанцирует словарь: `self.stats = {"processed_rows": 0, "time_ms": 0.0}`.
Во время выполнения метода `next()` операторы накапливают телеметрию.

Драйвер бенчмарка извлекает эту телеметрию, агрегирует ее и, с помощью библиотеки `matplotlib`, выводит подробные тепловые карты и графики. Это позволяет ответить не просто на вопрос "какая база быстрее", а **"в каком именно узле"** (при сканировании I/O или при вычислении хэшей для связи таблиц) произошла просадка производительности интерпретатора Python.
