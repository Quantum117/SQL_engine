# 2.8. Итераторная модель выполнения (Volcano Model)

Получив оптимизированный математический план, движок должен перевести его в алгоритмы обработки данных. Исторически сложилось, что первой и самой понятной моделью физического выполнения стала **Volcano Model** (модель итераторов), разработанная в 1990-х годах проектом Volcano.

Значение этой модели для архитектуры баз данных трудно переоценить. Ее главная инновация — **инкапсуляция**. Каждый узел плана является изолированным черным ящиком, выдающим строки по одной.

### 2.8.1. Принцип Pull-Based (Вытягивание)
В файле `executor/physical_operators.py` базовый класс `PhysicalOperator` реализует стандартный интерфейс итератора Python:

```python
class PhysicalOperator:
    def __iter__(self):
        return self

    def __next__(self) -> Dict[str, Any]:
        raise NotImplementedError
```
*Интуиция кода*: Корневой узел (например, `PhysicalProject`) не знает, откуда берутся данные — из CSV-файла, по сети или из кэша. Он просто вызывает `next(self.input)`. Запрос на чтение инициализируется *сверху*, и каскадом спускается к листьям (сканерам), которые забирают ровно одну строку из файла и отдают ее наверх. Это называется Pull-based execution.

### 2.8.2. Анализ реализации оператора (PhysicalFilter)

Рассмотрим физическую реализацию фильтра:

```python
class PhysicalFilter(PhysicalOperator):
    def __init__(self, input: PhysicalOperator, condition: Any):
        super().__init__()
        self.input = input        # Ссылка на дочерний узел
        self.condition = condition # Математический предикат (например, age > 18)

    def __next__(self) -> Dict[str, Any]:
        while True:
            row = next(self.input)                # Запрашиваем строку снизу
            if self.evaluate(self.condition, row):# Вычисляем предикат
                return row                        # Отдаем строку наверх, если подошла
```
*Интуиция памяти*: Обратите внимание на цикл `while True`. Оператор может запросить 100 строк у дочернего узла, прежде чем найдет ту, которая удовлетворит условию, и отдаст ее наверх. **Ключевое преимущество Volcano**: в оперативной памяти всегда находится ровно *одна* строка (`row`). Вы можете обрабатывать таблицы размером 1 ТБ на компьютере с 1 ГБ оперативной памяти (исключением являются блокирующие операции вроде `ORDER BY`, которые вынуждены материализовывать результат).

### 2.8.3. Скрытые затраты в интерпретаторе Python
Модель Volcano изящна, но в Python она страдает от колоссальной деградации производительности. 
В языках C/C++ вызов виртуальной функции `next()` обходится в несколько наносекунд. В Python каждый вызов `__next__` для *каждой* строки миллионной таблицы приводит к созданию новой рамки стека (stack frame), динамическому поиску методов и распаковке словарей `Dict[str, Any]`. Накладные расходы среды выполнения превышают 90% всего времени обработки запроса.

Это архитектурное узкое место привело к необходимости разработки параллельной парадигмы — **Векторизованного исполнения**, которая рассматривается в следующем разделе.
