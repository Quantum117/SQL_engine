# 2.6. Архитектура логического оптимизатора (RBO)

После этапа связывания созданный логический план семантически корректен, но крайне неэффективен. К примеру, базовая структура плана для запроса с блоком `WHERE` представляет собой `LogicalFilter`, обертывающий весь набор данных (`LogicalJoin` или `LogicalScan`). Это означает, что движок применит фильтр только *после* того, как считает все миллионы строк таблицы в память.

Очевидна необходимость трансформации плана. В движке эта задача возложена на **Rule-Based Optimizer (RBO)** — оптимизатор на основе набора детерминированных эвристик.

### 2.6.1. Паттерн "Стратегия" в оптимизации
Архитектура `LogicalOptimizer` в файле `planner/optimizer.py` базируется на паттерне "Стратегия". Оптимизатор представляет собой контейнер, который циклически применяет коллекцию независимых правил `OptimizationRule` к корню дерева.

```python
class OptimizationRule:
    # Базовый интерфейс для всех правил трансформации
    def apply(self, op: LogicalOperator, catalog: Catalog) -> LogicalOperator:
        raise NotImplementedError

class LogicalOptimizer:
    def __init__(self, catalog: Catalog):
        self.catalog = catalog
        self.rules = []

    def add_rule(self, rule: OptimizationRule):
        self.rules.append(rule)

    def optimize(self, plan: LogicalOperator) -> LogicalOperator:
        # Прогоняем план последовательно через все зарегистрированные стратегии
        for rule in self.rules:
            plan = rule.apply(plan, self.catalog)
        return plan
```

*Интуиция кода*: Архитектура открыта для расширения, но закрыта для модификации (Open-Closed Principle). Если мы захотим добавить поддержку CBO (Cost-Based Optimization) на основе статистики таблицы, нам не придется переписывать ядро парсера или исполнителя; мы просто создадим класс `CostBasedJoinReorderRule(OptimizationRule)` и добавим его в `self.rules` в конфигурации движка.

### 2.6.2. Рекурсивный обход дерева
Правила оптимизатора не просто проверяют верхний узел плана, они рекурсивно спускаются по нему к самым листьям. Для этого используется мета-программирование: каждая вариация правила вызывает `self.apply(op.input, catalog)` для дочерних узлов, гарантируя, что любая трансформация применяется снизу вверх или сверху вниз в зависимости от логики правила.

Такая модульность упрощает отладку: любое новое правило всегда можно отключить закомментировав одну строчку `add_rule`, чтобы визуально увидеть в Graphviz (как описано в разделе 2.11) разницу между планами.
