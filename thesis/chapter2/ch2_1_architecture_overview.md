# 2.1. Общая архитектура и конвейер обработки запросов

Концептуально, проектируемый SQL-движок построен как классический транслятор: он переводит высокоуровневый декларативный язык в низкоуровневые вызовы функций. Главная интуиция за архитектурой — строгое конвейерное разделение ответственности. 

### 2.1.1. Конвейер трансляции (The Pipeline Intuition)

Вместо того чтобы пытаться выполнить запрос сразу после его прочтения, движок разбивает процесс на несколько изолированных фаз. Каждая фаза принимает структуру данных от предыдущего этапа, обогащает или трансформирует ее, и передает дальше.

Архитектура системы включает четыре главных компонента:

```mermaid
graph TD
    SQL[Исходный текст запроса] -->|Парсер| AST[Абстрактное синтаксическое дерево]
    AST -->|Binder + Catalog| LP[Логический план]
    LP -->|Optimizer| OLP[Оптимизированный логический план]
    OLP -->|Physical Planner| PP[Физические операторы]
    PP --> Exec[Исполнение (Batch/Row)]
```

### 2.1.2. Анализ кода: Интеграция фаз планирования

Эта конвейерная логика явно отражена в классе `LogicalPlanner` (модуль `planner.py`), который связывает воедино дерево AST и метаданные каталога для получения базового плана:

```python
from parser.parser import SQLParser
from ast_nodes.nodes import SelectStatement
from .binder import Binder
from catalog.catalog import Catalog

class LogicalPlanner:
    def __init__(self, catalog: Catalog):
        self.catalog = catalog
        self.binder = Binder(catalog)

    def plan(self, ast) -> LogicalOperator:
        if isinstance(ast, SelectStatement):
            plan, _ = self.binder.bind_select(ast)
            return plan
        else:
            raise NotImplementedError("Only SELECT statements are supported currently")
```
*Интуиция кода*: `LogicalPlanner` получает на вход чистое абстрактное дерево (`ast`). Обратите внимание, что сам парсер не является зависимостью планировщика — планировщик оперирует только строго типизированными узлами (наследуемыми от `ASTNode`), а `Binder` использует `catalog` для проверки семантического смысла (существуют ли указанные таблицы в базе).

### 2.1.3. Разделение моделей выполнения
Наиболее архитектурно значимым решением в движке является позднее связывание физического плана. Оптимизированный план является абстрактным математическим выражением. Только на самом последнем этапе `PhysicalPlanner` принимает решение о том, как физически обработать этот граф. В зависимости от конфигурации, он инстанцирует либо классы `PhysicalOperator` (работающие по одной строке), либо `VectorizedOperator` (работающие с NumPy-пакетами).

Такое разделение (separation of concerns) гарантирует, что правила оптимизатора (например, `Predicate Pushdown`) пишутся единожды и автоматически работают для обеих парадигм исполнения.
