# Как защищать код: Синтаксический анализ и AST

**1. "Я не стал изобретать велосипед для проверки синтаксиса"**
> "Писать парсер SQL вручную — это огромная трата времени из-за сложных правил вложенности (например, скобки в скобках). Поэтому, как и в случае с лексером, синтаксический анализ у меня делает библиотека `Lark`. Я использую алгоритм **LALR(1)**, который отлично справляется с SQL."

**2. "Где правила грамматики? Это верхняя часть grammar.lark"**
> "Откройте `parser/grammar.lark`. Если внизу были регулярки для токенов (лексер), то сверху — правила грамматики. 
> Например, правило: 
> `select_statement: SELECT select_items FROM table_ref where_clause?`
> Парсер четко видит, что после SELECT должны идти колонки, потом FROM, потом таблица, а WHERE — опционально (знак вопроса). Библиотека сама следит, чтобы пользователь не написал `FROM SELECT` или не забыл скобку."

**3. "Как работает ast_builder.py?"**
> "Lark выдает сырое дерево (CST). В нем куча мусора: служебные токены, метки правил. Движку с этим работать неудобно. 
> Поэтому я написал `parser/ast_builder.py`. Этот класс проходит по сырому дереву и превращает его в красивые объекты Python (Data Classes), которые лежат в `ast_nodes/nodes.py`.
> 
> Например, если мы распарсили колонку `u.name`, `ast_builder.py` создаст красивый объект `ColumnRef(table='u', name='name')`. В итоге у меня получается **Абстрактное Синтаксическое Дерево (AST)**, где нет ничего лишнего, только суть запроса."
